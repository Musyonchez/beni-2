# 4.2.1 Phase 1: Research Requirements Gathering and System Design

## i. Research

The first activity in this phase is research, which lays the foundation for the USIU Campus Navigation System project. It involves reviewing both global and regional campus navigation platforms to identify best practices, strengths, and existing limitations. Global systems such as MIT WhereIs and NUS Campus Map with NextBus provide insights into features like interactive maps, route generation, and transport integration. On a regional scale, platforms such as the University of Cape Town campus maps and University of Nairobi wayfinding maps reveal the challenges of limited interactivity, basic layouts, and lack of real-time functionality. In addition, primary research will be conducted through surveys and interviews with USIU students, staff, and visitors to capture the unique navigation challenges faced on campus, such as finding classrooms, offices, or service centers in a timely manner.

## ii. Requirements Gathering

The second activity is requirements gathering, which transforms user needs into actionable system specifications. The functional requirements include searching for specific buildings, generating walking routes, viewing building details, and allowing administrators to update facility information. Non-functional requirements will focus on usability, system responsiveness, accessibility, scalability, and security. These requirements will be carefully analyzed, prioritized, and documented in a Software Requirements Specification (SRS) to serve as a guide for development. This ensures that the system is aligned with the realities of USIU's environment and addresses current limitations in campus orientation and accessibility.

## iii. Proposal

The third activity is the preparation of the project proposal, which consolidates the research findings and requirements into a structured plan. The proposal clearly defines the system objectives, scope, methodology, and expected outcomes. It also justifies the project by highlighting the gaps in existing solutions, particularly the absence of an interactive and real-time navigation system within USIU. By presenting how the system will address these gaps, the proposal ensures a shared understanding among stakeholders and provides a reference framework for monitoring progress throughout the project's lifecycle.

## iv. System Architecture and Design

The final activity in this phase is system architecture and design, which defines the structural and logical blueprint of the USIU Campus Navigation System. This includes creating use case diagrams to represent user interactions, data flow diagrams (DFDs) to illustrate the flow of information, and an entity-relationship diagram (ERD) to design the database structure. The system architecture will also specify the integration of the frontend (mobile/web application), the backend (API server), and the database (relational database such as PostgreSQL or MySQL), along with external services like Google Maps API for route generation. These design artifacts provide a roadmap for implementation, ensuring the system is technically sound, user-friendly, and aligned with project goals.
