# 4.2.3 Phase Three: Testing, Documentation, And Presentation

## i. Software Testing and Evaluation

The first step in this phase will involve unit testing, which will focus on verifying the smallest individual components of the system, such as frontend search forms, backend API endpoints, and database queries. Each unit will be tested independently to confirm that it performs its intended function correctly. For instance, the building search form will be tested to ensure that it retrieves accurate building details, while the database will be tested to confirm that it stores and retrieves location data correctly. Unit testing at this stage will help identify and fix errors early, reducing the cost of resolving issues later in the development process.

The next activity will be integration testing, which evaluates the interactions between the frontend, backend, and database. This ensures that the various system components work together seamlessly. Key scenarios, such as a student searching for a lecture hall, retrieving its location, and generating a walking route, will be tested end-to-end. Integration testing will confirm that data flows smoothly across the system and that the navigation experience is stable and reliable under different conditions.

Following this, system testing will be performed to evaluate the platform as a whole. This will include testing functional requirements (such as searching for buildings, displaying map locations, and generating walking paths) and non-functional requirements (such as response speed, usability, security, and scalability). The goal of this stage is to confirm that the USIU Campus Navigation System performs reliably under real-world conditions and aligns with the requirements set out in the Software Requirements Specification (SRS).

Finally, User Acceptance Testing (UAT) will be conducted to involve actual USIU students, staff, and visitors in evaluating the system. Test participants will be asked to perform tasks such as searching for specific classrooms, generating routes to offices, or locating facilities like the library or cafeteria. Their feedback will be collected and analyzed to refine usability, enhance the user experience, and validate that the system effectively addresses the navigation challenges it was designed to solve. By the end of this stage, the system will be validated against its objectives and prepared for deployment.

## ii. Project Write-Up

All research findings, methodologies, system designs, implementation details, and testing results will be documented in a formal academic report. This write-up ensures that the USIU Campus Navigation System is well-documented for academic evaluation and also serves as a reference for potential future enhancements, scaling, or integration with other university systems such as transport services, event management, or emergency response systems.

## iii. Final Presentation and Submission

The USIU Campus Navigation System will be presented to stakeholders, including lecturers, peers, and representatives of the student community. This final presentation will demonstrate the navigation challenges addressed, showcase the system's features such as interactive maps and route generation, and highlight its impact on improving orientation and accessibility at USIU. The demonstration will validate the project's success, confirm its contribution to solving real student challenges, and present opportunities for further expansion. The final report and system prototype will then be submitted for academic evaluation and assessment.
