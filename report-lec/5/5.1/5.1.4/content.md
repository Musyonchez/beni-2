# 5.1.4 Data Flow Diagram (DFD)

A Data Flow Diagram (DFD) is a visual representation of how data moves within a system. It focuses on the flow of information between processes, data stores, and external entities, showing how inputs are transformed into outputs through different system functions. Unlike flowcharts that show the sequence of operations, a DFD emphasizes what data is being transferred and where it flows, helping to understand the logical structure of a system. It is typically developed in levels—starting from a general overview (Level 0) and moving to more detailed diagrams (Level 1, Level 2, etc.) that break down specific processes.

## i. Level 0: Context Diagram

The Level 0 DFD, also known as the Context Diagram, represents the system as a single process and shows how it interacts with external entities. It provides a high-level view of data flow without going into detailed processes.

> [Figure 15: Level 0 Context Diagram]

Figure 15 above shows a level 0 context diagram, users serve as the main external entities who provide input to the system through location search queries, route requests, or specific building lookups. For example, a student may search for the quickest route to a lecture hall, or a visitor may request directions to administrative offices or parking areas. These requests are transmitted to the Campus Navigation System, which processes the input and returns corresponding navigation results such as route maps, estimated walking times, or alternative paths in case of restricted areas.

The system also interacts with administrators, who provide data inputs such as updated building locations, facility information, or notices about inaccessible areas. This ensures that users receive accurate and up-to-date navigation results. The administrator's role helps maintain the reliability and accuracy of the data used by the system.

Essentially, the Level 0 DFD focuses on depicting the exchange of data between users and the system without delving into how each function is executed internally. It shows the overall data movement, where external users send queries to the Campus Navigation System, and the system, in turn, outputs relevant route details and location information. This provides a clear and simplified visualization of the system's scope, emphasizing the input and output data flows that define the system's primary purpose — to help users navigate the university campus easily and efficiently.

## ii. Level 1: Detailed DFD

The Level 1 Data Flow Diagram (DFD) of the Campus Navigation System provides a more detailed representation of how the system functions internally by decomposing the single process shown in the Level 0 diagram into several interconnected sub-processes. This breakdown helps in understanding the logical flow of data within the system and how different components interact to deliver navigation services to users. Unlike the Level 0 DFD, which only shows external interactions, the Level 1 diagram dives deeper to explain the internal data transformations and the relationships between various processes, data stores, and external entities.

In this level, the main system process is divided into smaller, well-defined sub-processes such as User Authentication, Search Location, Generate Navigation Route, and Update Campus Data. Each of these processes performs a specific function that contributes to the overall efficiency and reliability of the Campus Navigation System. For instance, the *User Authentication* process ensures that only authorized users, such as students and staff, can access certain system features. The *Search Location* process allows users to input queries, such as searching for a particular classroom or facility, while the *Generate Navigation Route* process uses that input to determine the most optimal path within the campus. Meanwhile, Update Campus *Data* enables administrators to add or modify information such as new buildings, blocked routes, or event locations.

The diagram also highlights the data stores that support these processes — primarily the Campus Location Database and the User Information Repository. These repositories store essential information that allows the system to respond accurately and quickly to user requests. Data continuously flows between the users, the system processes, and these storage components, ensuring up-to-date navigation results and reliable user experiences.

Figure 16 above shows the Level 1 DFD further that emphasizes how these interconnected processes collaborate to produce smooth and effective system functionality. Similarly, when administrators update or modify campus data, the new information is stored and made immediately available to users for subsequent navigation requests.

This level of detail provides valuable insight into how the system's internal components support each other to maintain accuracy and real-time responsiveness. It showcases the efficiency of data flow within the Campus Navigation System, ensuring that the right information is delivered to the right user at the right time. The Level 1 DFD therefore serves as a bridge between the broad overview provided in Level 0 and the more granular processes detailed in Level 2, illustrating a well-structured and logically connected system that facilitates seamless movement and accessibility across the campus.

## iii. Level 2: Detailed DFD

The Level 2 Data Flow Diagram for the Campus Navigation System provides a detailed breakdown of the major processes that were introduced in the Level 1 DFD. While Level 1 focused on the broader interaction between the user, the system, and the main processes such as "User Authentication," "Location Search," and "Navigation Display," this Level 2 diagram expands those processes to show exactly how data moves within each function. It helps visualize how information is entered, processed, stored, and output in more specific detail.

The Level 2 DFD starts by examining the User Authentication process in depth. When a user logs in, their credentials are captured by the Input User Data sub-process. The information is then passed to the Validate Credentials sub-process, which checks it against the User Database. If the credentials match, the system allows access through the Grant Access sub-process and directs the user to the home dashboard. If not, an error message is sent back to the user interface for correction.

The next major process is the Search Location function. At this level, it is broken down into smaller steps such as Receive Search Query, Verify Search Input, Fetch Location Details, and Display Search Results. When the user types a name—like "Library" or "Administration Block"—the system first validates whether the name exists in the Location Database. Once verified, the relevant coordinates and building details are retrieved and displayed on the screen.

Another core process in this level is Generate Route. This part is expanded into Capture Starting Point, Retrieve Destination Coordinates, Calculate Optimal Path, and Display Navigation Route. The sub-processes show how the system determines the user's current location, retrieves the destination's coordinates from the Map Database, computes the shortest or most convenient route, and then visually presents it on the app's interactive map.

The Level 2 DFD also includes background processes like Update Map Information and Manage Feedback. The update function allows the system administrator to refresh location data or add new facilities through the Admin Panel, which is linked to the Location Database. The feedback process collects user input from the Feedback Form, processes it under Store Feedback Data, and saves it to the Feedback Repository for later analysis or system improvement.

> [Figure 17: Level 2: Detailed DFD]

Figure 17 above shows the level 2 DFD diagram. After developing the Level 2 DFD, the data flow of the entire Campus Navigation System became clearer and more transparent. Each interaction between the user, databases, and internal processes could now be traced step by step. For instance, when a student logs in and searches for a lecture hall, the diagram clearly shows how the request moves through authentication, validation, database retrieval, and finally to map display. This detailed flow ensures that no critical data movement is overlooked during implementation.

The diagram also highlights how internal components are interlinked. The User Database, Location Database, and Map Database interact constantly, supporting the smooth exchange of data. This interconnected structure ensures that users always receive up-to-date navigation details and accurate building information. Additionally, the flow from Generate Route back to the Display Results process ensures that users can see both their current position and the recommended route on one screen, improving usability.

After the diagram was completed, it became possible to analyze the efficiency of data handling within the system. The team could identify potential issues such as duplicated data paths or unnecessary processes and optimize them before coding. The Level 2 DFD therefore acted as a problem-prevention tool, reducing the risk of system errors during development. It also helped in finalizing how the databases would be structured and how the app's backend logic would interact with the user interface.

In conclusion, the Level 2 DFD for the Campus Navigation System provides a comprehensive view of the system's data movement and logic. It breaks down each main process—user login, location search, route generation, and feedback management—into clear, traceable sub-processes. Before the diagram, planning focused on identifying key system activities and their relationships; after the diagram, these ideas were refined into a working data structure ready for development. The Level 2 DFD ultimately serves as a bridge between conceptual design and actual programming, ensuring that every function of the app operates efficiently and consistently.
