# 4.2.3 Phase Three: Testing, Documentation, And Presentation

## i. Software Testing and Evaluation

The first step in this phase will involve unit testing of individual components — menu loading, cart calculations, wallet deduction logic, and Cloud Function behaviour. Each unit will be tested independently to confirm it performs its intended function correctly. Integration testing will follow, evaluating end-to-end scenarios such as a student placing a wallet order through to the staff screen updating and the FCM notification being received. System testing will then assess the platform as a whole against all 20 defined test cases, covering order flow, real-time synchronisation, pre-order cut-off accuracy, wallet atomicity, role-based access security, and performance under load. Finally, User Acceptance Testing (UAT) will be conducted with representative USIU students and cafeteria staff, who will perform realistic tasks — placing orders, scheduling pre-orders, topping up wallets, and managing incoming orders — and provide feedback on usability and reliability.

## ii. Project Write-Up

All research findings, system design artifacts, implementation details, and testing results will be documented in a formal academic report. This write-up ensures the USIU Cafeteria Ordering System is well-documented for academic evaluation and serves as a reference for any future enhancements, including M-Pesa wallet top-up integration, dietary filters, or a Kitchen Display System.

## iii. Final Presentation and Submission

The USIU Cafeteria Ordering System will be presented to stakeholders including lecturers and peers. The presentation will demonstrate the problems addressed, showcase the system's key features through a live demo — ordering, pre-scheduling, wallet management, and staff order handling — and highlight the impact on reducing peak-hour congestion and improving the dining experience. The final report and system prototype will then be submitted for academic evaluation.
