# 8.3 Challenges Encountered

Although the USIU Cafeteria Ordering System was successfully built and deployed, the development process presented several technical and architectural challenges that required significant problem-solving effort.

**Firebase Admin SDK incompatibility in Deno edge runtime.** The most significant technical challenge arose when implementing the Supabase Edge Functions. The Firebase Admin SDK uses gRPC internally for Firestore communication, but the Deno runtime used by Supabase Edge Functions does not support gRPC. Initial attempts to connect from the edge function to Firestore produced persistent connection errors that were not immediately obvious from the error messages. The resolution required explicitly configuring the Firebase Admin SDK with `db.settings({ preferRest: true })` to force the SDK to use the REST API instead of gRPC. This single configuration line was critical and is not prominently documented; identifying it required deep investigation of the Firebase Admin SDK internals and Deno runtime limitations.

**Firestore transaction design for atomic wallet operations.** Designing the wallet order placement flow to be fully atomic required careful structuring of the Firestore `runTransaction` call. An early implementation performed the balance check and balance deduction as separate Firestore writes, which created a race condition where two simultaneous order placements could both pass the balance check before either had decremented the balance. Migrating the logic into a single `runTransaction` block — which reads the current balance, verifies sufficiency, decrements it, and creates the order document in one atomic operation — resolved the race condition but required significant restructuring of the `FirestoreRepository` and `CartFragment` interaction.

**Fragment navigation and state preservation.** An early architectural decision to use `replace` fragment transactions for the bottom navigation caused the `CartViewModel` to reset whenever the user navigated away from the Cart tab. This meant items added to the cart were lost if the user browsed back to the Menu. Migrating from `replace` to a `hide`/`show` fragment management pattern in `MainActivity` and `StaffMainActivity` preserved each fragment's lifecycle and ViewModel state across tab switches, resolving the issue, but required a full refactor of the navigation setup.

**First-login forced password change for staff accounts.** Staff accounts are created by the admin via the Firebase Admin SDK with a temporary password. Ensuring that staff were forced to change this password on their first login required a `firstLogin` boolean field on the user document in Firestore and a `ChangePasswordActivity` that intercepted navigation immediately after login when the flag was true. Coordinating the flag reset (written after a successful `updatePassword` call) with the subsequent navigation to `StaffMainActivity` required careful sequencing to avoid race conditions between the Firestore write and the activity transition.

**FCM token management.** FCM tokens are generated per device and can be rotated by the platform. An early implementation stored the FCM token only at registration time, which caused notifications to stop arriving after a token refresh. The solution required implementing `onNewToken` in `MyFirebaseMessagingService` to write the refreshed token back to the user's Firestore document whenever FCM issued a new one. This ensured the `fcmToken` field on the `users` document was always current, regardless of when the rotation occurred.

**Multi-role security rule design.** Writing Firestore security rules that correctly handled three roles (student, staff, admin) across five collections without duplicating logic required several iterations. The initial rules were written with inline role checks per collection, leading to repetitive and error-prone conditions. Refactoring to use shared helper functions (`isAuth()`, `isOwner()`, `isStaff()`, `isAdmin()`, `isPrivileged()`) and testing all boundary conditions via the Firebase Console Rules Playground produced a clean, maintainable rule set that passed all three security test cases.
