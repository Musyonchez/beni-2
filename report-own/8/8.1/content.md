# 8.1 Introduction

This final chapter provides a comprehensive summary of the USIU Cafeteria Ordering System project, reflecting on its primary objectives, architectural decisions, implementation outcomes, and the technical challenges encountered during development. The application was built to address the persistent inefficiencies of the current manual cafeteria ordering process at USIU-Africa — long queues during peak hours, cash-only transactions, no advance ordering capability, and no real-time communication between students and cafeteria staff.

Background research during the requirements phase confirmed that no existing campus-specific tool addressed this problem holistically. Existing general-purpose food delivery applications did not integrate with the campus wallet system, could not be operated by cafeteria staff without smartphones, and lacked the institutional identity required for adoption within a university environment. These insights shaped the vision for the USIU Cafeteria Ordering System: a purpose-built, three-role Android application backed by a real-time cloud database, designed to digitise the entire cafeteria workflow from menu display to meal collection notification.

The system's primary aims were to eliminate the physical queue for common orders, provide students with an in-app digital wallet, enable advance pre-ordering around lecture timetables, and give cafeteria staff a structured, real-time order management interface. Secondary objectives included building an admin web panel for menu and user management, enforcing role-based data security through Firestore rules, and deploying all server-side logic on free-tier infrastructure to demonstrate viability without capital expenditure.

Development was executed in structured phases: requirements gathering, system design (UML, wireframes, class diagrams, data model), Android implementation across twelve incremental steps, Supabase Edge Function deployment, Firestore rules and index configuration, Next.js admin panel development, and rigorous manual testing. The iterative approach allowed issues such as the Deno gRPC incompatibility and the Firestore transaction atomicity design to be identified and resolved before they could affect the deployed system.

Testing confirmed that all fifteen test cases — functional, security, and performance — passed on a physical Android device against the live backend. User Acceptance Testing with peer participants validated that the interface was intuitive across all three roles. The system is deployed and operational, providing a working prototype that demonstrates the full end-to-end cafeteria ordering workflow and a clear technical foundation for future production deployment.
