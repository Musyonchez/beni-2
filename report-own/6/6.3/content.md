# 6.3 Backend Implementation

The backend architecture of the USIU Cafeteria Ordering System is built on Firebase's managed cloud services supplemented by Supabase Edge Functions for scheduled automation. Unlike a traditional server-client model requiring a dedicated application server, the system uses Firebase Firestore as both the real-time database and the source of truth for all application state, Firebase Authentication for identity management, and Firebase Cloud Messaging for push notifications. Supabase Edge Functions — serverless TypeScript functions running on the Deno runtime — handle the two automation tasks that require a server-side execution context: processing pre-order cut-offs on a schedule, and dispatching push notifications when an order status changes to "ready". The entire backend operates on free-tier plans with no ongoing operational cost.

## i. Firestore Data Model

The system's data is organised across five Firestore collections at the root of the database. The users collection stores one document per registered user, identified by the Firebase Auth UID. Each document contains the user's name, email, studentId, role ("student", "staff", or "admin"), walletBalance (present only for students), fcmToken (the device's current FCM registration token), and firstLogin (a boolean that is true until the user completes their first-login password change). Staff and admin users do not have a walletBalance field; their documents contain only identity and role data.

The menuItems collection holds one document per menu item, with fields for name, description, price, category ("breakfast", "lunch", or "dinner"), imageUrl, and available (a boolean availability flag). The orders collection stores customer order documents, each containing an embedded list of ordered items (name, price, quantity per item), the total amount, payment method, status, and an estimatedWaitMin value calculated at order creation time. The preOrders collection holds pre-order documents with a scheduledDate field (formatted "YYYY-MM-DD"), mealSlot, recurring flag, recurringDays list, and a status field that transitions through "scheduled", "confirmed", and "cancelled". The walletTransactions collection records every wallet credit and debit, with a staffId field on staff-initiated transactions providing an audit trail linking every balance change to the staff member who performed it.

Composite indexes are deployed for the five most performance-critical query patterns: student order history (userId ASC + createdAt DESC), staff order queue (status ASC + createdAt ASC), student pre-order listing (userId ASC + createdAt DESC), pre-order cut-off query (mealSlot ASC + scheduledDate ASC + status ASC), and wallet transaction history (userId ASC + createdAt DESC). These indexes were created via the Firebase Console and are required for Firestore to execute compound queries efficiently without throwing "index required" errors at runtime.

## ii. FirestoreRepository Pattern

All Firestore reads and writes in the Android application are routed through a single class, FirestoreRepository. No Fragment or ViewModel accesses the Firestore SDK directly; they call methods on the repository, which returns LiveData objects or accepts callbacks. This pattern enforces a strict separation between UI logic and data access, makes all database operations auditable in one place, and simplifies future testing and maintenance.

The repository exposes distinct methods for each data operation. listenToMenuItems() attaches a Firestore addSnapshotListener to the menuItems collection and pushes updates to a MutableLiveData list. listenToStudentOrders(userId) queries the orders collection filtered by userId, ordered by createdAt descending, and streams results via LiveData. listenToAllActiveOrders() serves the staff screen, querying for orders where status is in "pending", "preparing", or "ready". placeOrderWithWalletDeduction(order) executes a Firestore runTransaction that atomically deducts the order total from walletBalance and writes the order and wallet transaction documents in a single server-side operation. creditWallet(userId, amount, staffId) and deductWalletByStaff(userId, amount, staffId) both use runTransaction and embed the staffId in the resulting walletTransactions document. notifyOrderReady(orderId, userId) makes an HTTP POST to the Supabase notify-order-ready Edge Function after a staff member marks an order as "ready".

Figure 31 illustrates the centralised repository structure. The use of Firestore transactions in all wallet operations is a critical design decision: because Firestore transactions perform a read-then-write atomically on the server, concurrent wallet operations from multiple devices cannot result in an incorrect balance. If two operations attempt to modify the same balance simultaneously, Firestore retries the transaction automatically, ensuring consistency without any application-level locking.

## iii. MVVM Architecture — ViewModels and LiveData

The Android application follows the Model-View-ViewModel (MVVM) architectural pattern. ViewModels are Android Architecture Components that survive configuration changes such as screen rotation and persist across fragment switches within the same activity. Fragments observe LiveData objects exposed by ViewModels; when the data changes, the Fragment updates its UI automatically without polling.

CartViewModel holds the in-memory cart state as a MutableLiveData list of CartItem objects. It exposes addItem(), removeItem(), updateQuantity(), clearCart(), and setPaymentMethod() methods called by CartFragment. Because CartViewModel is scoped to the activity, the cart persists when the student navigates between tabs and is only cleared when the student explicitly places an order or logs out.

WalletViewModel maintains a real-time listener on the authenticated user's walletBalance field in Firestore. It exposes a LiveData of type Double — a balance observable that CartFragment uses to display the current balance on the payment toggle and enforce the insufficient-balance guard. Whenever the balance changes in Firestore — due to a top-up by staff or a deduction for an order — WalletViewModel's LiveData emits the new value and CartFragment's UI updates immediately without requiring a screen refresh.

OrdersViewModel manages the student's active and historical orders, using FirestoreRepository.listenToStudentOrders() to receive real-time status updates. Each status change emitted by the Firestore listener causes the RecyclerView adapter in OrdersFragment to update the affected order card's chip colour and label.

## iv. Supabase Edge Functions

Two Supabase Edge Functions implement the server-side automation that requires a trusted execution environment outside the Android app. Both functions are written in TypeScript and run on the Deno runtime. They authenticate incoming HTTP requests using a shared secret (FUNCTIONS_SECRET) passed as a Bearer token in the Authorization header. The secret is stored in Supabase project settings and is also embedded in the Android app's gradle.properties file, which is excluded from version control via .gitignore to prevent its exposure in source code.

The process-cutoff function is triggered by HTTP POST requests from cron-job.org at 07:00 UTC (10:00 AM East Africa Time) for lunch and at 14:00 UTC (5:00 PM East Africa Time) for dinner. The request body specifies the mealSlot. The function uses the Firebase Admin SDK to query the preOrders collection for documents where status is "scheduled", scheduledDate matches today's date, and mealSlot matches the target slot. For each matched document, it reads the student's walletBalance inside a Firestore transaction. If the balance is sufficient, it deducts the pre-order total, sets the pre-order status to "confirmed", creates a walletTransactions document, and sends a confirmation FCM notification. If the balance is insufficient, it sets the status to "cancelled" and sends a cancellation notification. For recurring pre-orders, the function creates a new preOrders document for the next weekly occurrence with status "scheduled", maintaining the automatic renewal chain without any student action.

The notify-order-ready function is called by the Android app via an HTTP POST immediately after the Firestore write that sets an order's status to "ready". The function reads the student's fcmToken from their users document and sends an FCM push notification informing the student that their order is ready for collection. This replaces the Firestore trigger approach that would require Firebase Cloud Functions on the Blaze plan; instead, the Android app takes responsibility for triggering the notification as a best-effort background call after the Firestore status update succeeds.

A critical implementation detail for both functions is the line db.settings({ preferRest: true }) called on the Firestore Admin SDK instance immediately after initialisation. The Deno runtime does not support gRPC, which is the default transport protocol for the Firebase Admin SDK's Firestore client. Without this setting, all Firestore calls from the edge functions fail silently. Setting preferRest to true instructs the Admin SDK to use the Firestore REST API instead, which is fully compatible with the Deno HTTP stack.

The function initialises the Firebase Admin SDK with service account credentials stored in Supabase secrets, applies the preferRest setting, then executes the batch processing loop.
